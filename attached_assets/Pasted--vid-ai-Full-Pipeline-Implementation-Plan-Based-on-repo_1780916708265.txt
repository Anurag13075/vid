# vid-ai — Full Pipeline Implementation Plan

> Based on repo audit: TanStack Start + React 18 + TypeScript frontend, Supabase auth/db, Bun runtime, Vite, Tailwind + Framer Motion. Frontend shell (landing, auth, topic input, editor mock) is complete. Everything below is what remains to be built.

---

## Free Stack Being Used

Every service in this pipeline is free. No paid APIs, no credit card required.

- Script generation — Groq (Llama 3.3 70B, free tier at console.groq.com)
- Voiceover — edge-tts (Microsoft Edge TTS, npm package, no API key, no limits)
- Footage — Pexels API (free) + Pixabay API (free) + Coverr (local pre-downloaded library, CC0)
- Music — Free Music Archive (pre-downloaded local library, CC licensed)
- SFX — Freesound API (free tier, freesound.org/apiv2/apply)
- Database + Auth + Storage — Supabase (free tier)
- Video assembly — FFmpeg (free, open source)

## Phase 1 — Database Schema (Supabase)

Before touching any pipeline code, the data model needs to be locked. Everything else hangs off this.

### Tables

**videos**
- id, user_id, title, status (queued / scripting / voicing / footage / assembling / done / failed), created_at, updated_at, output_url, thumbnail_url, duration_seconds, error_message

**video_sections**
- id, video_id, order_index, narration_text, visual_keywords (text array), section_type (broll / stat / graphic / intro / outro), key_point (nullable — triggers lower third), duration_ms

**video_assets**
- id, video_id, section_id, asset_type (footage / music / sfx / lower_third), source (pexels / pixabay / coverr / freesound / fma), external_id, local_path, duration_ms, license_confirmed (bool)

**render_jobs**
- id, video_id, stage, started_at, completed_at, error, retries

### RLS Policies
- Users can only read/write their own videos and assets
- Service role has full access for server-side pipeline operations

## Phase 3 — Script Generator (Groq)

### What It Does
Takes the video title, calls Groq with Llama 3.3 70B, returns a structured JSON object that drives every downstream stage. This is the most important module — the quality of the JSON schema directly determines the quality of the final video.

### Why Groq Over Claude Here
Groq's free tier is generous enough for script generation and Llama 3.3 70B is more than capable of writing documentary-style scripts. Groq is also significantly faster than most APIs — scripts generate in seconds. If quality becomes an issue on complex topics, swap to Claude Haiku (cheapest Claude tier) without changing anything else in the pipeline.

### Groq Prompt Strategy
- Model: llama-3.3-70b-versatile
- System prompt: you are a professional YouTube documentary scriptwriter
- Write in a narrative, engaging voice — not listicle unless the title calls for it
- Script must have a strong hook in the first 30 seconds, clear section breaks, and a CTA outro
- Must return pure JSON only — no markdown, no preamble, no explanation
- Also return a mood field on the root object: "dramatic", "uplifting", "neutral", or "tense" — used for music selection

### Output Schema Groq Must Return

Each video is broken into sections. Each section has:
- A narration string (what gets read aloud)
- An array of visual search keywords specific to that narration line — not the general topic
- A section type: broll, stat, graphic, intro, outro
- An optional key_point string — present only when this section deserves a lower third overlay
- An estimated word count so voiceover duration can be predicted
- An optional sfx boolean — true when a subtle sound effect would enhance the moment

### Section Type Rules
- If the narration contains a number, percentage, dollar figure, or measurable claim → section_type is "stat"
- If the narration is the opening hook → section_type is "intro"
- If it's the final CTA → section_type is "outro"
- Everything else → "broll"
- "graphic" is reserved for pure transition moments — insert 1–2 at natural breaks in long videos

### Visual Keywords Rule
This is the most critical detail. Keywords must describe the specific visual, not the topic. If narration says "Tesla's stock dropped 40% in a single trading session", keywords should be ["stock market trading floor", "red stock chart declining", "wall street traders worried"] — not just ["Tesla", "stocks"]. The footage agent uses these keywords verbatim to search Pexels and Pixabay.

### Target Length by Duration
- 8–10 min video → 18–22 sections
- 12–15 min video → 26–32 sections
- 20+ min video → 40–50 sections

## Phase 4 — Voiceover (edge-tts)

### What It Does
Takes each section's narration text, calls edge-tts, saves audio file per section, returns duration.

### Why edge-tts
It's a free npm package that uses Microsoft's Neural TTS voices — the same voices behind Azure Cognitive Services. No API key, no account, no rate limits in any practical sense. Quality is genuinely good for documentary narration.

### Best Voices for Documentary Style
- `en-US-AriaNeural` — warm, clear, female. Best default.
- `en-US-GuyNeural` — neutral, authoritative, male.
- `en-US-DavisNeural` — slightly deeper male voice, good for serious topics.

### Important Details
- Generate per-section, not the whole script at once — enables section-level retries and editing later
- Save files as `audio/{video_id}/{section_id}.mp3`
- Record exact `duration_ms` per section by reading the generated file's duration — assembler depends on this for clip timing
- Use a consistent voice for the entire video — store chosen voice on the video record
- Expose 3–4 voice options to the user in the UI before generation starts

## Phase 5 — Footage Agent

This is the most complex module. Handles sourcing, deduplication, fallback chain, and type-aware searching across three sources.

### Sources in Priority Order
1. Pexels API — best quality free footage, clean API
2. Pixabay API — large library, good fallback
3. Coverr local library — pre-downloaded CC0 clips, zero API calls, used when both APIs fail

### Deduplication Strategy
- Maintain a `usedClipIds: Set<string>` scoped to the entire video render job
- Before accepting any clip, check if its external_id is in the set
- If it is, skip and try the next result from the same API
- After accepting, add to set and persist to `video_assets` table

### Per-Section Logic

**For section_type = "broll"**
1. Search Pexels footage with visualKeywords
2. Filter: minimum 1080p, duration ≥ 8 seconds, not in usedClipIds
3. If Pexels returns nothing usable → search Pixabay with same keywords
4. If Pixabay fails → search Coverr local library index by keyword match
5. If all fail → flag section as missing_footage, assembler extends adjacent clip

**For section_type = "stat"**
1. Search Pexels/Pixabay with keywords emphasizing data/charts/graphs: "data visualization", "statistics chart", "infographic animation", "financial graph"
2. No motion graphics available in free sources — assembler will handle stat sections with FFmpeg text animation overlay on top of relevant broll

**For section_type = "graphic"**
1. Search for abstract, minimal, clean clips — "abstract motion background", "clean minimal loop", "light rays", "particle background"
2. These are 2–4 second clips used as visual breaks

### Pexels API Notes
- Base URL: https://api.pexels.com/videos/search
- Header: Authorization: {PEXELS_API_KEY}
- Params: query, per_page, orientation=landscape, size=large (1080p+)
- Response includes video_files array — pick the highest resolution file
- Free, commercial use allowed, attribution in description is good practice

### Pixabay API Notes
- Base URL: https://pixabay.com/api/videos/
- Params: key, q, video_type=film, per_page
- Response includes videos object with resolutions — pick large or full
- Free, commercial use allowed

### Coverr Local Library Setup
- Pre-download 300–500 clips from coverr.co before deploying
- Organize into folders by category: nature, city, technology, business, people, abstract, data
- Build a JSON index file: `[{ id, filename, category, tags[], duration }]`
- footageAgent searches this index by matching visualKeywords against tags array
- All Coverr content is CC0 — completely free for commercial use

### Music Selection
- Done once per video, not per section
- Read the mood field from the script JSON (dramatic / uplifting / neutral / tense)
- Pick one track from the local FMA library matching that mood folder
- FMA library setup: pre-download 30–50 tracks per mood from freemusicarchive.org, organize into mood folders, build a JSON index
- Store selected music path on video record

### SFX (Freesound)
- Only for sections where Groq tags `sfx: true` in the JSON
- Search Freesound API with section context keywords + "sound effect"
- Filter by license: cc0 only — ensures commercial use safety
- Keep a separate usedSfxIds set

## Phase 6 — FFmpeg Assembler

Takes all audio files and footage clips and produces a single MP4. Most complex module.

### Assembly Order of Operations

1. For each section, trim or loop footage clip to match voiceover duration + 0.5s buffer
2. Apply Ken Burns zoom/pan every 6 seconds within each clip
3. Apply xfade transition between every pair of adjacent clips
4. Overlay narration audio on the video timeline
5. For stat sections: overlay FFmpeg text animation showing the key stat figure
6. For sections with key_point: burn in lower third overlay
7. Mix BGM bed under narration (narration at 0dB, BGM at -12dB during narration, -3dB in intro/outro)
8. Apply SFX at tagged timestamps
9. Apply color LUT globally as final video filter
10. Final encode to H.264

### Zoom/Pan (Ken Burns) Logic
- Every clip is divided into 6-second chunks
- Each chunk gets a different zoom direction: zoom in, zoom out, pan left, pan right — alternated
- Parameters: zoom range 1.0 to 1.08, slow and smooth
- Implemented via FFmpeg zoompan filter

### Transitions
- Use xfade filter between clips
- Approved types: fade, slideleft, slideright, wipeleft, wiperight
- Duration: 0.5 seconds
- Randomize from approved list — never the same transition twice in a row

### Stat Section Text Animation
Since free sources don't have motion graphic templates, stat sections get an FFmpeg text overlay instead:
- Large bold white text centered on screen
- Dark semi-transparent background behind text
- Animated opacity: fade in over 0.5s, hold, fade out at end of section
- Font: use a clean sans-serif system font (Liberation Sans or similar)
- This replaces motion graphics for the free stack

### Lower Thirds
- Trigger: section has a non-null key_point value
- Show at second 1.5 of the section
- Duration: 3.5 seconds
- Built with FFmpeg drawtext: left-aligned, white text, colored accent line above
- Fade in 0.3s, hold, fade out 0.3s

### Color Grade
- Apply a LUT file to all footage for visual consistency
- Bundle 2–3 free LUT files in server/assets/luts/ (many free documentary LUTs available online under CC)
- Applied as the last video filter before encode

### Audio Mix
- Narration on track 1
- BGM on track 2, ducked during narration using FFmpeg volume envelopes
- SFX on track 3 mixed at appropriate timestamps
- Final mix normalized to -14 LUFS (YouTube recommended)

### Output Spec
- Resolution: 1920x1080
- Codec: H.264 (libx264), High profile
- Pixel format: yuv420p (required for YouTube)
- Audio: AAC 192kbps, 48kHz stereo
- Container: MP4
- Bitrate: 8–12 Mbps

## Phase 7 — Orchestrator & Job Queue

### Orchestrator Flow
Calls each stage sequentially, updates video status in Supabase at each step. If a stage fails it updates status to "failed" with error message and stops.

Stage order:
1. Set status → "scripting" → call scriptGenerator → save sections to DB
2. Set status → "voicing" → call voiceover for each section → save audio paths
3. Set status → "footage" → call footageAgent for each section → save asset records
4. Set status → "assembling" → call assembler → produce intermediate files
5. Set status → "exporting" → call exporter → upload final MP4 to Supabase Storage
6. Set status → "done" → save output_url and thumbnail_url

### Job Queue
Simple in-process queue for MVP — one video at a time per server instance. Structure it as a proper queue class so BullMQ can be swapped in later without rewriting the orchestrator.

### Retry Logic
- Each stage gets 3 retry attempts before marking the job failed
- Retries use exponential backoff: 5s, 15s, 45s
- Footage stage retries are per-section — one missing clip doesn't kill the whole job

### Real-time Status via SSE
- Frontend subscribes to /api/pipeline/:id/status
- Server sends SSE events at each stage transition
- Event payload: { stage, progress_pct, message }
- Frontend updates studio UI in real time

## Phase 8 — Storage

### Supabase Storage Buckets

**vid-ai-temp**
- audio/{video_id}/{section_id}.mp3 — per-section voiceover
- footage/{video_id}/{section_id}_{clip_id}.mp4 — downloaded clips (Pexels/Pixabay only — Coverr clips are local)
- sfx/{video_id}/{section_id}_sfx.mp3
- Cleaned up after successful render

**vid-ai-outputs**
- videos/{video_id}/final.mp4 — permanent until user deletes
- thumbnails/{video_id}/thumb.jpg

### Cleanup Job
After "done" status, clean up all temp files for that video_id after 24 hours.

## Phase 13 — Design Preservation Rules

The existing frontend looks clean and polished. Every new UI piece added during pipeline wiring must match it exactly. This is non-negotiable.

### Golden Rules

Never touch these existing files unless the only change is wiring real data in:
- Landing page components (hero, pricing, FAQ, navbar)
- Auth pages
- Any component already visually complete
- Global CSS / Tailwind config / font imports
- Framer Motion animation configs already in place

If a file already has styling, extend it — never rewrite from scratch.

### When Adding New Components

- Use only Tailwind classes and color tokens already present in the codebase — no new colors, no hardcoded hex values
- Match existing border-radius, spacing scale, and shadow style — look at adjacent components first
- Framer Motion: same easing curves, similar duration ranges as existing animations
- Typography: match existing font weights, sizes, line heights exactly
- Icons: use the icon library already in the project — don't add a second one

### Component-Specific Rules

**Progress / Status UI**
- Same card style, border, background as existing studio components
- Stage indicators: subtle — small dot or thin progress bar, not a giant spinner
- Error states: match the tone and color of any existing error UI

**Section List in Studio Editor**
- Match the visual rhythm of the existing mock exactly — don't change padding, gap, or card sizing
- Footage thumbnails: fixed 16:9 ratio, never stretch or overflow

**Video Player**
- Native HTML5 video element styled to dark editor theme
- No third-party player libraries unless already in the project

**Dashboard**
- Card-based layout consistent with existing patterns
- Status badges: small, pill-shaped, using only existing Tailwind config colors
- Never expose raw status strings to the user

**Modals**
- Use existing modal component if one exists — don't build a new one
- Backdrop blur, padding, border style must match existing modals

### Loading States
- Skeleton loaders for content areas — match the shape of the content they replace
- Inline spinners only for button actions — replace button text, keep same button size
- Never show a blank white flash

### Animations
Leave all existing Framer Motion animations exactly as they are. For new components, only:
- Fade in on mount (opacity 0→1, 200–300ms)
- Subtle slide up on mount (y: 8→0 with fade, 250ms)
- Scale on hover for interactive cards (scale 1→1.02, 150ms)

Nothing else — no bounces, no complex sequences.

### Pre-Commit Checklist
Before every frontend commit:
- Does it look identical to the design at 1280px?
- Did I accidentally change any existing component's layout or spacing?
- Are all text strings user-friendly — no raw API values, no camelCase exposed?
- Do loading states follow the skeleton/spinner convention?
- Does it work on mobile?
- Did I introduce any new color, font, or animation not already in the project?

## Key Risk Areas

**Footage quality gap** — Pexels and Pixabay are good but not as cinematic as Storyblocks. The Ken Burns motion and color grade in the assembler will close this gap significantly. Test early and often.

**FFmpeg complexity** — the assembler takes the longest to get right. Zoom/pan + transitions + overlays + audio mixing + LUT all in one filter graph is hard. Build and test each filter in isolation before composing.

**Render time** — a 10-minute video could take 15–30 minutes depending on server specs. Set expectations clearly in the UI. Consider a dedicated render server (Fly.io, Railway) separate from the API server.

**Groq rate limits** — free tier has per-minute token limits. Script generation is a single large request so it can hit limits if multiple users generate simultaneously. Add retry with backoff and surface a friendly "busy, retrying" message to the user.

**Freesound OAuth** — Freesound uses OAuth 2.0 for API access, not just an API key. The server needs to handle the client credentials flow. Test this early — it's more involved than Pexels/Pixabay which are simple header auth.

**Coverr local library size** — 300–500 clips is a lot of disk space. Pre-download selectively by category based on the most common video topics your client expects to generate.
